# Phase 2: 設計（Design）フェーズ定義

## ゴール
Phase 1で確定した要件に基づき、システムのアーキテクチャを設計し、テスト戦略を策定する。

## 参加ロール
- **PM（プロジェクトマネージャー）**: 議論の進行管理、ゲート判定
- **Architect（アーキテクト）**: システム設計、技術選定
- **QA（品質保証）**: テスト戦略策定、設計レビュー

## 議論の進め方
1. Architectが要件定義書を元に、アーキテクチャの初案を提示する
2. QAがテスト可能性の観点から設計をレビューする
3. PMが要件との整合性を確認する
4. チーム全体でトレードオフを議論し、最適な設計を合議する
5. PMがゲート通過を判断する

## ゲート基準（Phase 2 → Phase 3 進行条件）
以下がすべて満たされた場合、PMはゲート通過を宣言する：
- [ ] システム構成（コンポーネント分割）が定義されている
- [ ] 技術スタックが選定理由付きで決定されている
- [ ] データモデルが設計されている
- [ ] 主要APIのインターフェースが定義されている
- [ ] テスト戦略（テスト種別、範囲、優先度）が策定されている
- [ ] 設計が全ての要件を満たしていることが確認されている

## 成果物

- `test_strategy.md` — テスト戦略書

## 入力
